# 5000K - Manifest

> We build software for people who want to own their tools, not rent access to them.

We believe good software should give people more control, not less. It should be possible to run it yourself, understand how it works, adapt it to your needs, and leave when you want to. Without losing your data, your workflow, or your independence.

We are not interested in building dependency traps. We don't accept software that only works as long as someone else keeps the gate open, products that manufacture complexity to justify themselves, or systems that quietly turn their users into tenants.

We build software that stays close to its purpose. That means tools small enough to understand, direct enough to use, and solid enough to keep. We would rather make a sharp tool than a vague platform. We would rather solve one real problem well than wrap a simple idea in layers of optional machinery.

We value simplicity, but not as decoration. For us, simplicity is a practical quality. A simple system is easier to host, easier to debug, easier to trust, easier to change, easier to fix, easier to replace, and easier to keep alive for years.

We do not treat complexity as a virtue. Complexity is a cost, and it has to justify itself. If something can be made clearer, smaller, or more direct without losing what matters, that's usually the better design.

We believe good software should be understandable from the outside and workable from the inside. Its moving parts should be visible. Its data should not be held hostage. Its configuration should not feel like a ritual. Its integration points should not require surrender.

We like software that uses plain formats, simple interfaces, and boring technology. We trust tools that make fewer assumptions. We trust systems that can explain themselves. We trust designs that leave room for the user's own infrastructure, workflow, and judgment.

We care about developer experience, but not as an abstraction layer that hides reality. Good developer experience means the path is short, the behavior is legible, and the system does what it says. Setup should be quick, integration straightforward, and maintenance proportional to the problem being solved.

We care about security and privacy the same way: as things that have to be built into the shape of the tool, not applied on top as branding. Security should not demand needless ceremony. Privacy should not require blind trust.

A 5000K project should leave a user more independent than it found them. It should reduce reliance on closed systems, hidden state, and accidental complexity. It should preserve exits where possible. It should respect the user's data, time, and attention. It should justify every major layer it introduces.

Not every good project belongs in 5000K. A project fits here when it reflects this attitude: it increases ownership, maintains data control, stays understandable, justifies the complexity it introduces, and helps people do real work without becoming the work itself.

We are builders, and we build for our kind. But more than that, we build for anyone who wants their software to stay theirs.

# 5000K - Manifest

> We build software for people who want to own their software and their data.

We believe good software gives people more control, not less. You can run it yourself, understand how it works, adapt it to your needs, and leave whenever you want to. At no point in this process are you losing your data, your workflow, or your independence.

User count never was and never will be a good metric to measure how good a piece of software is. Bad software traps users and creates migration walls so it can be satisfied with only being good enough to not bother leaving.

We are not interested in building dependency traps. We don't accept software that only works as long as someone else keeps the gate open, products that manufacture complexity to justify themselves, or systems that quietly turn their users into tenants.

We build software that stays close to its purpose. That means our software stays small enough to understand what it does exactly. We would rather make a sharp tool than a vague platform. We would rather solve one real problem well than wrap a simple idea in layers of optional machinery.

We value simplicity. For us, simplicity is a practical quality. A simple system is easier to host, easier to debug, easier to trust, easier to change, easier to fix, easier to keep alive for years and also easier to replace.

Complexity is a cost, and it has to justify itself. If something can be made clearer, smaller, or more direct without losing anything relevant to the core usecase, that will usually the better design.

We believe good software should be understandable from the outside and workable from the inside. Its moving parts should be visible. Data may not be held hostage. Configuration may not feel like a ritual. Integration points need to be well defined and purposeful.

We believe that good software uses plain formats, simple interfaces, and should be as boring as possible. We trust tools that make fewer assumptions. We trust systems that can explain themselves. We trust designs that leave room for the user's own infrastructure, workflow, and judgment.

Good developer experience means the path is short, the behavior is obvious, and the system does what it says. Setup should be quick, integration straightforward, and maintenance proportional to the problem being solved.

We care about security and privacy the same way: as things that have to be built into the shape of the tool, not applied on top as branding. Defaults need to be as secure as possible. Requiring blind trust is a clear anti-pattern.

Not every good project belongs in 5000K. A 5000K project should leave a user more independent than it found them. It should reduce reliance on closed systems, hidden state, and accidental complexity. It should preserve exits where possible. It should respect the user's data, time, and attention. It should justify every major layer it introduces.

We are builders. We carry independence in our heart. We build for our kind. We build for anyone who wants their software to stay theirs.
